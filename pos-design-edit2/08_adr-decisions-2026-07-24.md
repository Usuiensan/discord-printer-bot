# 08. ADR判断・推奨案（2026-07-24）

## 1. 扱い

本書は、要件定義v0.5に対する2026-07-24の利用者判断と、説明を求められた項目への設計推奨を記録する。

- `Accepted`: 利用者判断として実装へ適用する。
- `Proposed`: AI推奨。利用者承認までは本番仕様として確定しない。
- `Research`: 調査中で、判断に追加情報が必要。
- 要件v0.5と異なる判断は、原文を黙って読み替えず「要件改訂が必要」と明示する。

## 2. Accepted

### ADR-P004 任意値引きのイベント初期値

- 状態: `Accepted`
- 初期値は率上限100%。取引合計0円まで許可し、負数は禁止する。
- 責任者承認機能と理由入力機能は当面実装しない。
- 実装は `manual_discount_policies` を単純化し、初期版では `max_rate_bps=10000` だけを持つ。将来拡張用の未使用UIや必須入力を作らない。
- 要件差分: v0.5 6.6、17章、AC-10には承認・理由設定があるため、次版で「当面対象外」に改訂する必要がある。

### ADR-D007 商品・価格・在庫の所属単位

- 状態: `Accepted`
- 商品コード、名称、価格、税率・税区分、原価、販売状態、在庫はすべてevent別にする。
- 同じ商品コードを別eventで再利用できる。同一event内では一意とする。
- 過去取引はevent商品を参照し直さず、取引時点の値をsnapshot保存する。
- 実装は全イベント共通の `products` を正本にせず、`event_products` が商品コードを含む正本となる。

### ADR-D008 税込価格・端数・値引き配賦

- 状態: `Accepted`
- 顧客表示、レジ表示、レシート明細の頒布価格は税込価格とする。
- 消費税額はレシート全体を税率別に集計してから、各税率につき1回だけ計算する。
- 初期端数処理は切捨て。既存要件どおり切捨て・四捨五入・切上げを次営業から切替可能にする。
- 値引きは対象行へ整数円で配賦する。複数税率にまたがる取引共通値引きは、値引前税込金額の比率で配賦し、余りは「小数部が大きい順、同値は税率、行番号順」の最大剰余法で決定的に配る。
- 返品は元取引snapshotの金額・税率別集計を反転し、現在の価格・税設定で再計算しない。
- 国税庁は、適格請求書の消費税額等について一請求書につき税率ごとに1回の端数処理を求め、切上げ・切捨て・四捨五入の選択自体は任意としている。

参考:

- https://www.nta.go.jp/taxes/shiraberu/taxanswer/shohi/6371.htm
- https://www.nta.go.jp/law/tsutatsu/kihon/shohi/01/08.htm

### ADR-D009 取消・返品・外部決済返金

- 状態: `Accepted`
- 元レシートに対する部分返品UIは実装しない。返品は元取引の全商品・全支払を反転する「全部返品」を基本とし、残す商品は新規会計として再登録する。
- レシートがない返品は、商品コード、数量、返金方法、担当者、理由を入力する独立返品として記録できる。
- 元取引、在庫移動、支払を削除・上書きせず、反転記録を追加する。
- 「複数回返品専用」の業務機能や受入条件は設けない。ただし会計整合性のため、同じ販売数量・支払額を超える累積返金はDBトランザクション内で防ぐ。
- 外部決済端末の取消結果はPOSが推測せず、担当者が `completed/failed/unknown` を記録する。

### ADR-D010 見本在庫

- 状態: `Accepted`
- 見本在庫という区分、残数、専用movement、専用CSV、専用画面を実装しない。
- 展示に使った商品も販売可能在庫と同じで、販売時は通常販売とする。
- 汚損した場合だけ既存の破損・廃棄movementを使用する。
- 要件差分: v0.5 6.2、6.9、14.4、CL-07から見本専用表現を削除する必要がある。

### ADR-D011 R18確認

- 状態: `Accepted`
- event初期policyは担当者による `appearance`（容貌確認）だけを必須とする。
- 顧客画面の自己申告は補助機能であり、無効化または顧客画面故障時に省略できる。利用する場合も、容貌確認または身分証目視の代替にはしない。
- 保存するのは確認方式 `appearance/id_visual`、結果、確認者、取引、日時。自己申告を使用した場合はその実施結果も保存する。
- 身分証の種類、番号、氏名、住所、生年月日、画像は保存しない。

### ADR-D012 レシートの担当者表記

- 状態: `Accepted`
- event設定で `manager_no` または `display_name` の一方を選択する。両方印字は初期対象外。
- 初期値は要件5.1と整合する `manager_no`。
- `display_name` はシステム登録名であり、法的氏名を要求しない。ペンネーム、ハンドルネームを許可する。
- 取引snapshotには印字済み文字列を保存し、後のスタッフ名変更で過去レシートが変わらないようにする。

### ADR-D015 現金箱モード変更

- 状態: `Accepted`
- 営業中の `register/shared` 変更は禁止し、締め後に予約して次のbusiness day開始時から適用する。
- 過去のcash ledgerを移し替えず、各business dayへ使用モードをsnapshot保存する。

### ADR-D016 取り置き不足

- 状態: `Accepted`
- 販売可能数量を超える取り置きを許可する。
- 作成前に不足数と作成後の利用可能数を警告表示し、担当者の明示続行を要求する。
- 在庫・取り置きを自動補正せず、負数および関連する取り置きIDを監査・締め警告へ残す。

### ADR-D017 無料配布の顧客画面

- 状態: `Accepted`
- 購入取引に紐づく無料配布では、対象行の金額が0円へ切り替わる短い表示を行い、その後通常の会計表示へ戻す。
- 管理画面から行う会計外の無料配布では、顧客画面へ内部理由や担当者情報を送らず、待機表示「いらっしゃいませ」を維持する。
- アニメーション実装が難しい初期版は、待機表示の継続を許容する。金額や売上へ無料配布を混入させないことを優先する。

## 3. Accepted: データエクスポート方針

### ADR-P005 無料配布理由と会計ソフトCSV

- 状態: `Accepted`（2026-07-24）
- 利用者判断: 無料配布の理由入力機能は当面実装しない。
- 最小記録は商品、数量、日時、担当者、event、原価snapshot。売上・支払・税率別売上には含めない。
- 現時点では特定の会計ソフトを使用しないため、会計ソフト固有adapterは初期実装しない。
- 管理画面からevent、business day、日時範囲を指定し、運用中でも随時canonical export bundleを生成できるようにする。
- exportは正本DBを変更せず、同じschema version・範囲・DB状態から同じ業務データを再生成できる。
- 要件差分: v0.5 2章、6.7、17章には理由区分・任意メモがあるため、次版で当面対象外へ改訂する必要がある。

調査結果:

- 弥生会計は決められた列順の弥生インポート形式による仕訳CSV/テキスト取込に対応する。再取込時に重複警告がない製品もあるため、POS側のexport IDと取込済み管理が必要。
- freee会計はfreee形式CSVを取り込め、他社会計ソフト向けにも弥生、マネーフォワード、PCA等のテンプレート仕様を持つ。
- マネーフォワード クラウド会計は仕訳帳CSVのサンプルを提供し、取込時に項目を対応付ける。
- したがって、単一の「会計ソフト共通CSV」は採用しない。初期版は会計ソフトから独立したcanonical dataを保存・出力し、将来採用製品が決まった時点で変換adapterを追加する。

初期export bundle:

1. UTF-8 BOM付きRFC 4180 CSVとして、取引、明細、値引配賦、税率別集計、支払、返品・取消、在庫移動、無料配布、取り置き、現金台帳、締めを別ファイルで出力する。
2. `manifest.json` に `schemaVersion`、export ID、event、抽出範囲、生成日時、各ファイルの行数・SHA-256を保存する。
3. 金額は整数円、時刻はUTC ISO 8601、商品コードと取引番号は文字列、税率はbasis pointsで出力する。
4. CSVの全行に内部UUIDとevent IDを含め、後日の変換・突合・重複検出を可能にする。
5. export操作自体は監査へ記録するが、業務データに「出力済み」フラグを書かず、何度でも再出力できる。
6. 将来会計ソフトを採用したとき、その製品のサンプルCSVを基に変換adapterとgolden testを追加する。

公式資料:

- 弥生会計 Next: https://support.yayoi-kk.co.jp/subcontents.html?page_id=29610
- 弥生会計の記述形式: https://support.yayoi-kk.co.jp/faq_Subcontents.html?page_id=18586
- freee形式CSV: https://support.freee.co.jp/hc/ja/articles/202847920
- freeeの各社テンプレート仕様: https://support.freee.co.jp/hc/ja/articles/14687912396825
- マネーフォワード仕訳帳CSV: https://biz.moneyforward.com/support/account/guide/import-books/ib01.html

会計ソフト名、勘定科目、部門、税区分、日次集約方式は、製品導入時に変換adapter側で決める。POS正本とcanonical exportの実装をブロックしない。

## 4. Accepted: 推奨案の採用

### ADR-D013 自動値引きの重複

#### 何を決めるのか

同じ商品2冊が「2冊で100円引き」と「3冊で20%引き」の両方に数えられるなど、同じ1個の商品数量を複数ルールが同時に消費してよいかを決める。率値引きを順番に掛ける方式では、1,000円に10%引き、さらに20%引きを行うと、順番や基準額によって結果が変わり得る。

#### 推奨案

- 状態: `Accepted`（2026-07-24）
- 商品1個分の数量tokenは、自動値引きルール1回だけに割り当てる。同じtokenへの重複適用は禁止する。
- 複数ルールは、異なる数量tokenまたは異なる商品行を使う場合だけ同じ取引で併用できる。
- 率値引きは「そのルールへ割り当てた値引前税込金額」を基礎に1回だけ計算する。別値引き後の残額へ順番に掛けない。
- 全候補から最終支払額が最小の組合せを選び、同額はpriority、rule ID順とする。

これは実装が比較的単純で、二重割引と説明困難な順序依存を防ぎながら、別商品への複数キャンペーンは許容できる。

### ADR-D014 取引番号と営業単位

#### 何を決めるのか

内部IDとは別にレシートへ印字する番号を、いつ1へ戻すか、複数レジでどう一意にするか、締めを再開した際に番号を再利用するかを決める。

#### 推奨案

- 状態: `Accepted`（2026-07-24）
- `business_day` は、eventで「営業開始」してから「締め」るまでの1営業単位。日付をまたいでも同じbusiness dayとしてよい。
- 表示番号はbusiness day内で全レジ共通の6桁連番 `000001`。DB transaction内で採番する。
- canonical表記は `YYYY-MMDD-NNNNNN`、レシート上は `YYYY-MMDD-NNN-NNN`（例 `2026-0724-123-456`）とする。
- 欠番を許容し、採番済み番号を詰め直さない。取消・返品も元番号を再利用しない。
- 締め再開後は同じbusiness dayの続き番号を使う。
- 部分締めは初期版で実装せず、締め対象はbusiness day全体とする。
- 内部参照はUUIDv7を使い、表示番号の変更や桁あふれから分離する。

この案は4レジ間の番号衝突がなく、問い合わせ時に短く読み上げられ、再開処理も単純になる。

### ADR-D018 保持期限後の保守

#### 何を決めるのか

保存期間が過ぎたデータについて、何を消すか、backup内のコピーをどう扱うか、誤削除からどう復旧するかを決める。行単位削除は取引・在庫・監査の参照関係を壊しやすく、SSDでは単純なファイル削除だけで復旧不能を保証しにくい。

#### 推奨案

- 状態: `Accepted`（2026-07-24）
- アプリ内の自動削除・行単位削除・匿名化は初期版で実装しない。
- event終了時にDB、CSV、manifest、hashを一つの暗号化archiveへ固定し、event単位で保管する。
- 保持年数は法務・税務判断としてeventに記録し、システムが年数を決めない。複数データ種別が同居する場合は最長期間を適用する。
- 期限到来時はサーバー停止、直前backup、対象archiveと全backupコピーの一覧確認、責任者承認、削除、削除証跡の保存という保守runbookで処理する。
- 復旧不能性が必要ならarchiveを固有鍵で暗号化し、全コピー削除後に鍵も破棄する。媒体の物理廃棄要否は運用側が決める。

最も実装が簡単なのは「削除機能を持たず手動archive管理」である。最も安全なのは、それに暗号化・鍵破棄・複数人確認を加える案である。

### ADR-D019 会場LANの通信保護

#### 何を決めるのか

HTTPでは、同じWi-Fi上の第三者や不正アクセスポイントに通信を観測・改変されると、PIN入力、session cookie、会計操作が漏えい・悪用され得る。HTTPSは通信を暗号化するが、インターネットの公開証明書を使えない会場LANでは、各iPad/PCへローカルCAを信頼させる導入作業が必要になる。

#### 推奨案

- 状態: `Accepted`（2026-07-24）
- 本番はローカルCAで署名したHTTPSを採用する。固定hostnameと必要なら固定IPを証明書SANへ含め、reverse proxyでTLS終端する。
- 端末追加runbookに、CA証明書配布、iPadでの明示的な信頼設定、証明書fingerprint確認、接続試験、端末紛失時の解除を含める。
- cookieは `Secure`、`HttpOnly`、`SameSite=Strict`。Origin検査、CSRF token、短いsession期限、端末pairing、ログのsecret redactionを併用する。
- Wi-Fiは専用SSID、WPA2/WPA3、強い事前共有鍵、client isolationを基本とし、サーバーfirewallで必要portだけ許可する。
- HTTPはloopback開発環境だけを既定許可とする。初回実機検証で一時使用する場合は、外部接続なしの隔離LAN、明示したrisk acceptance、当日限りの資格情報を条件にする。

最も簡単なのはHTTPだが、PINと会計sessionを扱う本番推奨にはしない。HTTPSの導入作業を受け入れる案が最も無難である。

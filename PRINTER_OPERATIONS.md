# レシートプリンター操作・運用の知見

この文書は、Discord から Epson TM-T70II などの ESC/POS レシートプリンターを操作するときに得られた知見を、別の印刷機能にも再利用できる形でまとめたものです。

対象は「文字列を印刷できたか」だけではありません。印刷データの生成、転送、プリンター状態、紙への出力、カット、運用ログまでを一つの処理として扱います。

## 最初に押さえる原則

プリンター操作では、次の4段階を分けて確認します。

1. 設定値が意図した値になっている
2. アプリが意図した ESC/POS バイト列を生成している
3. そのバイト列が意図したプリンターへ送信されている
4. 実機が意図した紙面・動作を返している

たとえば、アプリがフルカット命令を正しく生成しても、プリンターのカッター機構が部分カット専用なら紙は完全分離しません。コード上の分岐、生成バイト、実機結果は別々の証拠です。

「API呼び出しが成功した」「デバイスファイルへの書き込みが完了した」だけで印刷成功とは判断しません。最終的には紙面、カット状態、エラー表示、サービスログを確認します。

## このシステムの印刷経路

印刷内容はアプリ内で ESC/POS バイト列に変換され、選択したバックエンドへ送られます。

| バックエンド | 経路 | 主な用途 |
| --- | --- | --- |
| `linux-usb` | Debian Bot → `/dev/usb/lp0` など | 通常運用。構成が単純 |
| `bridge` | Debian Bot → HTTP → Windows印刷ブリッジ → Windowsプリンター | OPOSやWindowsドライバーが必要な場合 |
| `windows` | Windows Bot → Windowsプリンター | Bot自体をWindowsで動かす場合 |

新規構成では、USBパススルーと双方向通信が安定するなら `linux-usb` が扱いやすい構成です。Windows固有のOPOS状態取得が必要な場合だけ `bridge` を選びます。

バックエンドを切り替えるときは、Botの設定だけでなく、状態取得の経路も同時に切り替わることに注意します。

## ESC/POSデータ生成

### 初期化と状態の持ち越し

ESC/POSプリンターは、配置、太字、文字倍率、行間などの状態を後続データへ持ち越します。1ジョブの先頭で初期化し、装飾を使った後は明示的に標準状態へ戻します。

途中に画像、QRコード、バーコード、カットを挟む場合も、次のテキストへ配置や画像用設定が漏れていないか確認します。

### 紙幅と印字幅

TM-T70IIの58mm運用では、このプロジェクトは通常 `PRINT_WIDTH_DOTS=384` を使います。ただし「58mmなら必ず384 dots」ではありません。機種、解像度、ドライバー、余白設定によって実際の印字可能幅は変わります。

幅を決めたら、少なくとも次を同じ値から計算します。

- 文字の折り返し幅
- 左右2列のレイアウト
- 画像の縮小幅
- QRコードとバーコードの最大幅
- プレビューのキャンバス幅

プレビューだけ合う、または実機だけ合う状態にしないことが重要です。

### 文字コードと画像フォールバック

TM-T70IIの日本語文字印刷では、プリンターが直接扱える文字コードとUnicodeの差を考慮します。

このプロジェクトの `TEXT_RENDER_MODE=auto` は、ジョブ内に直接印字できない文字が1文字でもあれば、そのジョブの全テキストを画像へ統一します。一部分だけ字体や行高が変わることを避けるためです。

モードの考え方は次のとおりです。

| 値 | 動作 |
| --- | --- |
| `text` | 可能な限りプリンター文字として送る |
| `auto` | 非対応文字があればジョブ全体を画像化する |
| `image` | 常にジョブ全体を画像化する |

画像化には次のトレードオフがあります。

- Unicode文字を安定して表現できる
- レイアウトを再現しやすい
- 転送データ量と印刷時間が増える
- フォント、二値化、行高の調整が必要になる

連続する文字行は1枚のラスタ画像へ結合し、QRコード、バーコード、途中カット、独立画像の位置だけで分割すると、余計な紙送りや隙間を抑えられます。

画像化文字には `TEXT_IMAGE_DITHER_MODE=threshold` の完全二値化が向いています。写真には `IMAGE_DITHER_MODE=ordered` などの誤差表現が向きます。同じ変換設定を無条件に使い回すと、文字がぼやけたり写真の階調が消えたりします。

### 円記号

日本語設定のプリンターでは、CP932の `0x5C` が紙面上で円記号として印字される場合があります。このプロジェクトではUnicodeの `¥`（U+00A5）をこのプリンター上の等価表現として扱います。

そのため、`¥` だけを理由にジョブ全体を画像化しない設計にしています。ただし、別機種や別の国際文字セットへ流用するときは、実機で `0x5C` の見え方を再確認してください。

## 画像印刷

画像は次の順序で正規化します。

1. EXIF回転を反映する
2. 透明部分を白背景へ合成する
3. 印字可能幅へ縮小する
4. グレースケール化する
5. 用途に合う二値化またはディザリングを行う
6. ESC/POSラスタ形式へ変換する

ESC/POSラスタでは、横方向を8ドットずつ1バイトへ詰めます。画像幅が8の倍数でない場合も、1行あたりのバイト数は切り上げます。

確認時は元画像ではなく、実際にプリンターへ送る直前の二値画像を見るのが有効です。このプロジェクトが画像モード使用時に印字画像をDiscordへ添付するのは、その観測性を確保するためです。

## カット操作

現行実装のカット命令は次の対応です。

| `CUT_MODE` | 送信バイト | 意味 |
| --- | --- | --- |
| `none` | カット命令なし | 紙送りのみ |
| `partial` | `1B 69` (`ESC i`) | 部分カット |
| `partial3` | `1B 6D` (`ESC m`) | 別形式の部分カット |
| `full` | `1D 56 41 00` (`GS V 65 0`) | フルカット要求 |

途中カットでは、文字がカッター位置へ掛からないよう、`CUT_FEED_LINES` の行数だけ前後に紙を送ります。

### TM-T70IIの物理的制約

TM-T70IIは紙を1点つないで残す部分カット機構です。`full` のバイト列が正しく送信されても、完全分離にならない実機があります。

フルカットが必須要件なら、ソフトウェアのカット命令を増やすのではなく、完全切断に対応するプリンターまたはカッターユニットを選びます。同じ命令を繰り返し送る方法は、紙詰まりやカッター負荷につながるため採用しません。

### 自動カットを確実に止める設定

現行コードでは `CUT_AFTER_PRINT` は `CUT_MODE` の既定値を決めるために使われます。`CUT_MODE` を明示した場合はそちらが実際の最終カットへ使われます。

したがって、現在の実装で自動カットを確実に止める設定は次です。

```env
CUT_AFTER_PRINT=false
CUT_MODE=none
```

`CUT_AFTER_PRINT=false` だけに依存せず、起動ログと生成バイトからカット命令がないことも確認します。

## プリンター状態の取得

### Linux USB

`linux-usb` では、`DLE EOT` のリアルタイムステータス要求を使い、プリンター、オフライン、エラー、用紙の状態を読みます。

```bash
npm run linux:status
```

デバイスは状態確認時に読み書き両用で開く必要があります。USBパススルーが書き込みだけを許していると、印刷できても状態要求はタイムアウトします。

```env
LINUX_PRINTER_DEVICE=/dev/usb/lp0
LINUX_STATUS_ENABLED=true
LINUX_STATUS_TIMEOUT_MS=1000
```

状態取得だけが不安定な場合は、切り分けのため一時的に `LINUX_STATUS_ENABLED=false` にできます。ただし、紙切れやカバーオープンを印刷前に検出できなくなります。恒久対策はUSB双方向通信を直すことです。

### WindowsとOPOS

Windowsでは、通常のプリンターキュー状態に加え、EPSON OPOS ADK for .NETからカバー、用紙切れ、ニアエンドを取得できます。

OPOSはデバイスを排他的にClaimするため、別プロセスが使用中だと状態確認自体が失敗します。印刷処理との競合は、一時的なビジーと物理エラーを分けて扱います。

### 状態の分類

再試行してよい状態と、人が直すまで止める状態を分けます。

再試行候補:

- 前ジョブの処理中
- 印刷中止処理中
- 一時的なI/O待ち
- 別プロセスによる短時間の占有

即時に人の確認が必要:

- オフライン
- 用紙切れ
- カバーオープン
- 紙詰まり
- 一時停止
- サービス要求

再試行は回数と待ち時間に上限を設けます。このプロジェクトでは試行ごとに待ち時間を増やし、利用者へ最初の再試行を通知します。

## 「印刷成功」の判定

次の証拠を強い順に扱います。

1. 実際の紙面とカット結果を確認した
2. プリンターがジョブ完了後に正常状態へ戻った
3. デバイスまたはブリッジへの全バイト送信が完了した
4. アプリがバイト列を生成できた

`write()` の完了は、OSがデータを受け取ったことを示すだけで、紙への定着を保証しません。また、状態確認で一時的な `EBUSY` が出てもジョブ自体は印刷済みの場合があります。エラー文字列だけで再送すると二重印刷になるため、紙面とプリンター状態を先に確認します。

## 障害切り分け

| 症状 | 最初に確認すること | 次の確認 |
| --- | --- | --- |
| `/dev/usb/lp0` がない | Proxmox hostの `usblp`、USBパススルー | 接続し直し後のデバイス番号 |
| `EACCES` | 実行ユーザー、`lp`グループ、device mode/gid | systemdの `User=` |
| 状態確認がタイムアウト | USBが双方向か、デバイスパス | タイムアウト延長、一時的な状態監視無効化 |
| 印刷は出たが `EBUSY` | 実際の紙面、同時に状態監視したプロセス | 二重送信せず次ジョブで再確認 |
| 文字化け | 文字コード、国際文字セット | `TEXT_RENDER_MODE=auto` の画像結果 |
| 文字の下端が欠ける | ラスタ画像の上下余白、行高 | 実送信直前の二値画像 |
| 写真が真っ黒・真っ白 | ディザ方式、しきい値 | 文字用設定が写真へ漏れていないか |
| QRが読めない | モジュールサイズ、余白、紙幅 | 実機で複数端末から読取 |
| バーコードが切れる | データ長、モジュール幅 | 384 dots内に収まるか |
| `full` でも一部つながる | 送信バイト、実機型番 | カッター機構の仕様 |
| Bot更新後も挙動が古い | systemdのMainPIDと起動時刻 | 配置先のcommit、journal |
| Windowsブリッジが失敗 | HTTP status/body、token一致 | Windows側キューとOPOS Claim |

## 安全な検証手順

設定変更や機種変更後は、小さな検証ジョブから段階的に進めます。

1. プリンター型番、紙幅、接続先デバイスを記録する
2. 状態確認だけを実行する
3. 初期化、ASCII 1行、改行だけを送る
4. 日本語と `¥` を送る
5. 左・中央・右揃えと2列レイアウトを確認する
6. 小さな二値画像を印刷する
7. QRコードとバーコードを実際に読み取る
8. 十分に紙送りして部分カットを1回試す
9. 通常のDiscord投稿を印刷する
10. 用紙切れ、カバーオープン、復旧後の再試行を確認する

各段階で、入力、設定値、生成バイト、ログ、紙面写真を対応付けると、後から再現しやすくなります。

raw ESC/POSは、サイズ制限と許可ユーザーを設けた状態で使用します。NVメモリ変更、ファームウェア操作、デバイス設定変更、過度なカッター駆動を含む命令は、通常の印刷テストへ混ぜません。

## デプロイ後の確認

コードの更新と、稼働中プロセスの更新は別です。このリポジトリでは `tools/update-debian.sh` が、fast-forward更新、依存関係、構文確認、テスト、systemd再起動、状態とjournal確認を順番に行います。

更新後は少なくとも次を確認します。

```bash
systemctl show discord-printer-bot \
  --property=MainPID \
  --property=ActiveEnterTimestamp \
  --property=ExecMainStartTimestamp

journalctl -u discord-printer-bot -n 50 --no-pager -o cat
```

最後にテスト投稿を行い、Discord上の完了表示だけでなく、実際の紙面とカットを確認します。

## 実装を変更するときの確認項目

- 通常投稿、参加通知、記号印刷、ReceiptLineの全経路で設定が一貫している
- `!preview` と実機印刷が同じ幅・折り返し・2列内容を使う
- ジョブ末尾カットと途中カットを別々にテストしている
- `none`、`partial`、`partial3`、`full` の生成バイトをテストしている
- 非対応文字が1文字あるジョブ全体の画像化をテストしている
- `¥` が不要な画像フォールバックを起こさない
- 文字画像の設定が写真画像へ漏れない
- 再試行後の二重印刷リスクを検討している
- 状態監視無効時の動作も確認している
- rebaseや競合解消後に全テストを再実行している

実装箇所は主に次のファイルです。

- `src/escpos.js`: ESC/POS生成、画像ラスタ化、カット
- `src/printer.js`: バックエンド選択、状態確認、送信、再試行
- `src/discordContent.js`: Discord投稿から印刷ジョブへの変換
- `src/receiptLineContent.js`: ReceiptLineの描画
- `src/rasterText.js`: 文字画像の生成
- `src/config.js`: 印刷設定
- `tools/linux-printer-status.js`: Linux USB状態の単体確認
- `tools/linux-rawprint.js`: Linux USBへのrawデータ送信
- `tools/update-debian.sh`: Debian本番更新と確認
